ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ЧАСТЬ 1): СИСТЕМНАЯ АРХИТЕКТУРА, КЛАССЫ И ЯДЕРНАЯ ЛОГИКА

1. Концептуальное определение и назначение системы

ZeroSeg XDR (Extended Detection and Response) — это высокопроизводительный программный комплекс, функционирующий на стыке User Space (пользовательское пространство) и Kernel Space (пространство ядра). Система предназначена для обеспечения непрерывного мониторинга безопасности конечных точек и принудительного исполнения политик комплаенса (Compliance Enforcement).

В отличие от классических межсетевых экранов (Firewalls), которые работают по статическим спискам доступа (ACL), ZeroSeg XDR реализует динамическое реагирование: система самостоятельно сопоставляет состояние сетевых сокетов с базой юридических требований (ФСТЭК, ГОСТ, ISO) и принимает решение об изоляции узла без участия оператора, основываясь на заданных профилях риска.

2. Модульная структура и Объектно-Ориентированная модель

Программа спроектирована на языке ISO C++17. Выбор языка обусловлен необходимостью прямого манипулирования структурами памяти сетевого стека и минимальными накладными расходами на исполнение (Zero-overhead principle).

2.1. Класс-контроллер ZeroSegEDR

Центральный объект системы, инкапсулирующий все защитные механизмы.

Инкапсуляция данных: Все чувствительные данные (активные блокировки, IP-адреса, дескрипторы сокетов) находятся в private секции, доступ к которым осуществляется через строго определенные методы, что исключает возможность подмены правил в процессе работы.

Жизненный цикл: При инициализации (конструктор) происходит автоматическая сборка «цифрового отпечатка» системы (OS Fingerprinting). Это критично для кроссплатформенности: программа определяет, использовать ли Winsock2 (Windows) или POSIX Sockets (Linux) еще до начала первого сканирования.

2.2. Модуль Audit Subsystem (Сенсорный уровень)

Реализует алгоритм Non-blocking TCP Port Polling.

Системные вызовы: Используется функция socket(AF_INET, SOCK_STREAM, 0).

Оптимизация задержек: Для предотвращения блокировки основного потока выполнения при сканировании "фильтруемых" (DROP) портов, применяется setsockopt с параметром SO_SNDTIMEO.

Математическая модель: Таймаут установлен на уровне $T = 100\text{ms}$. Это позволяет просканировать стандартный набор из 1000 критических портов менее чем за 2 минуты, в то время как стандартные инструменты могут тратить на это до 30 минут из-за ожидания подтверждения TCP-рукопожатия.

3. Compliance Knowledge Base (CKB) — База знаний комплаенса

Это уникальный компонент, превращающий сканер уязвимостей в систему аудита ИБ. Реализован как std::map<int, ComplianceInfo>, где ключом является номер порта, а значением — структура, содержащая:

Legal Reference: Ссылка на конкретный пункт приказа (например, ФСТЭК №239 для КИИ).

Standard ID: Идентификатор международного стандарта (ISO/IEC 27001).

Threat Vector: Описание вектора атаки (например, «Уязвимость протокола SMB к атакам типа Remote Code Execution через эксплойт EternalBlue»).

Это позволяет программе генерировать отчеты, понятные не только техникам, но и юридическим отделам или внешним аудиторам.

4. Enforcement Engine (Механизм подавления уггов)

Этот модуль отвечает за физическое изменение конфигурации сетевого экрана ОС.

4.1. Логика Active Rejection (REJECT vs DROP)

ZeroSeg XDR намеренно отказывается от политики DROP в пользу REJECT.

Механика DROP: Пакет просто игнорируется. Злоумышленник видит задержку (Timeout), что явно указывает на наличие фаервола.

Механика REJECT (ZeroSeg): Ядро отправляет пакет TCP RST (Reset) или ICMP Port Unreachable.

Результат: Инструменты сканирования злоумышленника (например, nmap) помечают порт как closed, а не filtered. Это создает иллюзию того, что сервис на машине физически отсутствует, скрывая сам факт наличия системы защиты.

4.2. Метод Stateful Session Killing (NF_CONNTRACK)

Одна из самых сложных и важных функций. В современных ОС Linux существует таблица conntrack (Connection Tracking).

Проблема: Если пользователь запустил VPN-туннель до активации защиты, простые правила iptables не разорвут уже установленную сессию.

Решение: ZeroSeg XDR при обнаружении активного Shadow IT (VPN/Telegram) выполняет системный вызов для полной очистки таблицы состояний: sudo conntrack -F. Это принудительно сбрасывает все текущие TCP/UDP сессии. При следующей попытке передачи пакета ядро обнаружит новые правила блокировки и запретит соединение.

5. Технология блокировки Shadow IT (Telegram/VPN)

Блокировка современных сервисов требует работы не только с именами, но и с инфраструктурой.

DNS Resolving: Программа выполняет резолвинг доменов (например, vpngate.net) в реальном времени.

CIDR Masking: Поскольку Telegram использует сотни динамических IP, ZeroSeg блокирует их целыми блоками. Используются префиксы /20 (4096 адресов) и /22 (1024 адреса), охватывающие основные дата-центры Telegram (например, диапазон 149.154.160.0/20).

6. Кроссплатформенная абстракция (HAL)

Программа реализует уровень абстракции от оборудования (Hardware Abstraction Layer):

Linux: Взаимодействие с цепочками INPUT и OUTPUT через netfilter.

Windows: Инициализация библиотеки ws2_32.lib. Блокировка через интерфейс netsh advfirewall (взаимодействие с Windows Filtering Platform).

FileSystem: Использование кроссплатформенной библиотеки std::filesystem (введенной в C++17) для управления логами, что исключает ошибки при работе с разными типами слешей в путях (/ или \).

ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ЧАСТЬ 2): ЭКСПЛУАТАЦИЯ, ВЕРИФИКАЦИЯ И ОТКАТ
1. Регламент развертывания и установки (Deployment Flow)
Для обеспечения корректной работы ZeroSeg XDR на целевом узле, процесс установки должен следовать строгому алгоритму предварительной проверки зависимостей.

1.1. Системные требования и зависимости
Программа требует низкоуровневого доступа к сетевым интерфейсам и таблицам маршрутизации.

Linux (Target): Ядро версии 4.15+. Необходимы установленные пакеты iptables и conntrack-tools.

Windows (Target): Права локального администратора. Библиотека ws2_32.lib (линкуется при сборке).

Среда сборки: Компилятор с полной поддержкой стандарта ISO C++17 (GCC 9+, MSVC 2019+).

1.2. Процесс компиляции (Build Pipeline)
Использование стандарта C++17 критично для работы модуля std::filesystem, который отвечает за интеллектуальное управление отчетами.

Команда сборки (Linux):

Bash
# Флаг -lstdc++fs необходим для корректной линковки файловой системы в старых версий GCC
g++ -std=c++17 src/zeroseg_combined.cpp -o zeroseg_edr -lpthread -lstdc++fs
Команда сборки (Windows MinGW):

Bash
g++ -std=c++17 src/zeroseg_combined.cpp -o zeroseg_edr.exe -lws2_32
2. Алгоритм интеллектуальной отчетности (Smart Reporting)
Система отчетности ZeroSeg XDR построена на принципе атомарной инкрементальной записи. Это исключает риск потери данных предыдущих аудитов при повторных запусках.

2.1. Механизм Atomic Versioning (v1, v2, vN)
Программа реализует рекурсивный алгоритм проверки существования файлов в директории reports/.

Шаблон имени: REP_[IP]_[DATE]_v[N].txt.

Логика поиска: Функция generateReportPath() проверяет наличие файла с индексом v1. Если файл существует, индекс увеличивается, и проверка повторяется до тех пор, пока не будет найдено свободное имя.

Цель: Создание непрерывной цепочки доказательств (Chain of Custody) для ИБ-аудиторов. Позволяет сравнить состояние системы «до» и «после» применения политик защиты.

2.2. Структура технического отчета
Каждый сгенерированный файл содержит:

Metadata: Версия ОС, текущий IP-адрес, точное время начала и конца аудита.

Findings: Список обнаруженных открытых портов с привязкой к ФСТЭК №239 или ISO 27001.

Actions: Перечень примененных блокировок (IP, домены, CIDR-диапазоны).

Sync Status: Результат работы скрипта синхронизации с удаленным репозиторием.

3. Методология верификации защиты (QA & Monitoring)
После активации защитных механизмов, администратор обязан провести верификацию работы ZeroSeg XDR с использованием независимых инструментов.

3.1. Техническая проверка через iptables
Самый надежный способ подтвердить блокировку на уровне ядра:

Bash
sudo iptables -L OUTPUT -v -n
Критерий успеха: Если в колонке pkts (пакеты) и bytes (байты) значения отличны от нуля в строках, созданных ZeroSeg, — значит, трафик физически перехватывается ядром.

3.2. Живой мониторинг через tcpdump (Сниффер)
Для подтверждения отправки флагов RST (Active Rejection):

Bash
sudo tcpdump -i any host [TARGET_IP]
Критерий успеха: Появление пакетов с флагом [R.]. Это доказывает, что система не просто блокирует трафик, а активно имитирует закрытый порт, вводя злоумышленника в заблуждение.

4. Регламент аварийного отката (Rollback & Failsafe)
Для исключения ситуаций «самоблокировки» или потери доступа к критическим узлам (например, по SSH), в ZeroSeg XDR реализован трёхуровневый механизм отката.

Уровень 1: Интерактивный (Soft Rollback)
В конце каждой сессии программа предлагает пользователю подтвердить сохранение блокировок. При выборе n (отмена), программа итерирует по вектору active_blocks и вызывает команду удаления правила:
sudo iptables -D [CHAIN] [RULE_INDEX]

Уровень 2: Полный программный сброс (Hard Reset)
Если работа программы была прервана некорректно (Crash), и доступ к сети нарушен, администратор должен выполнить:

Bash
# Полная очистка всех цепочек и сброс состояний conntrack
sudo iptables -F && sudo conntrack -F
Это мгновенно возвращает сетевой стек в состояние «Default-Accept».

Уровень 3: Windows-специфичный откат
Использование системной утилиты netsh для сброса конфигурации Брандмауэра:
netsh advfirewall reset

5. Последовательность выполнения (Operational Sequence)
Для минимизации ошибок оператора, программа следует жестко заданному Workflow:

Инициализация: Проверка прав суперпользователя. Если UID != 0, выполнение прерывается.

Инвентаризация: Сбор данных об интерфейсах и текущем IP.

Аудит: Опрос портов (21, 22, 80, 445, 3389) с таймаутом 100мс.

Интерактив: Запрос подтверждения блокировки для каждого выявленного риска.

Enforcement: Применение правил iptables + conntrack -F (сброс сессий).

Reporting: Запись отчета в reports/.

Sync: Автоматический вызов sync_reports.sh для Git-синхронизации.

6. Условия стабильности и ограничения системы (Caveats)
DNS-резолвинг: Блокировка по доменным именам (telegram.org, vpngate.net) требует наличия связи с DNS-сервером. В полностью изолированных сетях блокировка будет работать только по портам.

Конфликты с Docker/Kubernetes: Поскольку ZeroSeg использует iptables, возможны конфликты с правилами NAT, создаваемыми контейнеризацией. Рекомендуется использовать на чистых узлах (Endpoints).

IPv6: Текущая версия фокусируется на трафике IPv4. Для полной защиты рекомендуется отключить IPv6 в системе или добавить модуль ip6tables
